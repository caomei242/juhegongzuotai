# 草莓工作台导航指挥台设计

日期：2026-06-16

## 目标

建立一个本地网页应用，作为用户聚合多个网页系统的「草莓工作台」。

第一版不是普通书签页，而是一个可自由配置的个人工作指挥台：

- 聚合主业、客户、内容、财务、实验项目和个人常用网页。
- 支持拖拽配置分组和链接卡片。
- 自动检测链接是否可访问。
- 手动维护每个网页系统的业务状态、备注和今日动作。
- 让用户每天打开后能快速判断：哪些系统要处理、哪些链接异常、哪些网页可以直接进入。

## 已确认决策

- 产品名固定为「草莓工作台」。
- 左上角必须显示「草莓工作台」。
- 第一版所有可见界面文案使用简体中文。
- 主界面采用「可拖拽系统桌面 + 今日动作侧栏」。
- 第一版做成本地网页应用，不做纯静态 HTML。
- 浏览器负责界面、拖拽、编辑和打开链接。
- 本地服务负责保存配置和自动检测链接健康。
- 数据真源先使用项目内 JSON 文件。
- 当前主力工作机先作为运行机器。
- 架构必须为后续迁移部署预留空间，不能把数据路径、端口和机器环境写死。
- 第一版不做登录、多用户协作、云同步、浏览器插件和自动生成今日动作。

## 视觉方向

最终视觉方向为中文工作台界面：

- 左侧为分组导航。
- 中间为可拖拽网页系统卡片画布。
- 右侧固定为今日动作、异常链接和当前链接编辑面板。
- 整体风格克制、清晰、偏工作台，不做营销式首页。
- 色彩以浅色中性底、石墨文字为主，用蓝色作为命令强调，用绿色、黄色、红色表达健康状态。
- 卡片和面板圆角不超过 8px。
- 不使用装饰性光斑、巨型 hero、卡片套卡片或过度阴影。

## 主界面结构

### 左侧分组栏

左上角固定显示：

```text
草莓工作台
```

默认分组建议：

- 主业系统
- 客户管理
- 内容工具
- 财务账本
- 实验项目
- 个人常用

分组栏能力：

- 新增分组。
- 重命名分组。
- 删除分组。分组内还有链接时，默认阻止删除，需要用户先移动链接或明确确认连同链接删除。
- 调整分组排序。
- 点击分组后筛选中间画布。
- 显示每个分组下链接数量。
- 显示每个分组的异常链接数量。

### 顶部工具栏

顶部工具栏包含：

- 全局搜索框：搜索系统名、域名、备注和今日动作。
- 添加链接。
- 添加分组。
- 编辑布局开关。
- 全量检测。
- 健康状态筛选。
- 业务状态筛选。

搜索和筛选只改变当前显示结果，不改变数据排序。

### 中间系统卡片画布

中间区域按分组展示网页系统卡片。

每张卡片字段：

- 系统名称。
- 链接地址。
- 域名或简短路径。
- 所属分组。
- 健康状态。
- 最后检测时间。
- 手动业务状态。
- 备注。
- 今日动作。
- 排序位置。
- 是否置顶。

卡片操作：

- 打开链接。
- 编辑链接。
- 删除链接。
- 拖拽排序。
- 拖拽到其他分组。
- 标记置顶。
- 修改业务状态。
- 修改备注。
- 修改今日动作。

编辑布局开启时，显示拖拽手柄、分组拖放区域和排序反馈。编辑布局关闭时，界面优先服务快速打开和查看状态。

### 右侧固定栏

右侧固定栏包含三个区域。

第一块是「今日动作」：

- 显示所有填写了今日动作的链接。
- 可按优先级或分组排序。
- 可一键打开对应链接。
- 可清空或编辑今日动作。

第二块是「异常链接」：

- 显示健康状态为异常或不可用的链接。
- 显示最后检测时间。
- 支持重新检测单个链接。
- 支持进入编辑面板修正地址。

第三块是「当前链接」：

- 当用户选中一张系统卡片时，展示该链接详情。
- 可编辑分组、地址、业务状态、备注、今日动作和检测频率。
- 保存后立即更新中间卡片和 JSON 数据。

## 健康检测

健康检测由本地服务执行。

状态定义：

- 正常：请求成功，状态码可接受。
- 异常：目标可访问但返回重定向过多、非预期状态码或响应过慢。
- 不可用：连接失败、超时或 DNS 失败。
- 未检查：新链接或尚未完成检测。

检测规则：

- 支持手动「全量检测」。
- 支持单个链接重新检测。
- 支持按配置间隔自动检测。
- 检测结果写入数据文件。
- 检测不能覆盖用户手动维护的业务状态、备注和今日动作。

第一版健康检测不尝试登录网页，也不判断业务功能是否可用。它只判断链接入口是否基本可访问。

## 手动业务状态

业务状态由用户维护。

默认状态：

- 正常使用
- 待处理
- 重点关注
- 暂停使用
- 待迁移
- 已废弃

状态列表需要可配置，但第一版可以先通过配置文件维护，不要求在界面里管理状态字典。

## 数据设计

数据真源放在独立 `data/` 目录。

建议结构：

```text
data/
  workbench.json
  health-checks.json
  backups/
```

`workbench.json` 保存：

- 分组列表。
- 链接列表。
- 排序位置。
- 用户手动字段。
- 状态字典。
- 基础设置。

`health-checks.json` 保存：

- 链接 id。
- 最近检测时间。
- 最近健康状态。
- 最近响应耗时。
- 最近错误信息。
- 连续失败次数。

备份规则：

- 每次重要写入前保留一个最近备份。
- 用户可以手动导出完整 JSON。
- 用户可以导入 JSON 恢复或迁移。

## 可迁移部署

第一版先运行在当前主力工作机，但按可迁移部署设计。

必须避免：

- 把绝对数据路径写死在代码里。
- 把端口写死在代码里。
- 把机器名写死在代码里。
- 把检测频率写死在代码里。
- 把访问口令写死在代码里。

可配置项：

- 服务端口。
- 数据目录。
- 健康检测频率。
- 健康检测超时时间。
- 是否启用访问口令。
- 访问口令。
- 是否允许局域网访问。

配置来源：

- 第一版优先使用 `.env` 或本地配置文件。
- 默认配置要能开箱即用。
- 迁移到另一台机器时，应能通过复制项目目录和 `data/` 目录完成迁移。

后续部署方向：

- Intel Mac 内网常驻。
- NAS 或小主机常驻。
- Docker 部署。
- 云服务器部署。

第一版不需要实现 Docker，但目录结构、配置和数据位置不能阻碍后续加入 Docker。

## 错误处理

用户编辑数据时：

- 必填字段缺失时阻止保存。
- 链接地址格式明显错误时提示。
- 删除分组前，如果分组内还有链接，默认阻止删除，需要用户先移动链接或明确确认连同链接删除。
- 导入 JSON 前先校验结构。
- 导入失败不能破坏当前数据。

健康检测时：

- 单个链接检测失败不能影响其他链接。
- 全量检测需要展示进度或检测中状态。
- 检测超时后标记为不可用，并记录错误原因。
- 服务重启后保留上次检测结果。

数据写入时：

- 写入前生成备份。
- 写入失败时提示用户，并保留内存中的当前编辑状态。

## 测试重点

第一版至少验证：

- 新增链接后能保存并刷新显示。
- 编辑链接后 JSON 数据正确更新。
- 删除链接后不残留在今日动作和异常链接中。
- 新增、重命名、删除分组行为正确。
- 拖拽排序后刷新页面仍保持顺序。
- 链接跨分组拖拽后数据正确。
- 搜索和筛选可以组合使用。
- 今日动作侧栏只显示有今日动作的链接。
- 异常链接侧栏只显示异常或不可用链接。
- 健康检测失败不会影响页面使用。
- 导出 JSON 后可重新导入恢复。
- 修改数据目录配置后仍能启动并读取数据。
